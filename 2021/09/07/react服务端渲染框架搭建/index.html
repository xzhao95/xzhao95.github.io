react服务端渲染框架搭建

发表于 2021-09-07 分类于 学习笔记 阅读次数： Valine：

前言

之前工作中也曾使用过服务端渲染，但都是直接使用的一键式的脚手架，对于底层还是一知半解的，所以想自己重新从零搭建，在实践中了解整套逻辑。本文主要记录与整理，实践过程中查阅过的资料以及实践的过程。

服务端渲染

在实践之前主要还是要先了解什么是服务端渲染，以及为什么要用服务端渲染。

定义

顾名思义，服务端渲染就是在浏览器请求页面URL的时候，服务端将我们需要的HTML文本组装好，并返回给浏览器，这个HTML文本被浏览器解析之后，不需要经过 JavaScript 脚本的执行，即可直接构建出希望的 DOM 树并展示到页面中。这个服务端组装HTML的过程，叫做服务端渲染。

服务端渲染的利弊

  • 利于SEO

对于客户端渲染，是十分不利于爬虫的，因为返回的HTML是一个空壳，它需要执行JavaScript脚本之后才会渲染出真正的页面。

  • 白屏时间更短

相对于客户端渲染，服务端渲染在浏览器请求URL之后已经得到了一个带有数据的HTML文本，浏览器只需要解析HTML，直接构建DOM树即可。而客户端渲染，需要先得到一个空的HTML页面，这个时候页面已经进入白屏，之后还需要经过加载并执行JavaScript、请求后端服务器获取数据、JavaScript渲染页面几个过程才可以看到最后的页面。特别是在复杂应用中，由于需要加载JavaScript脚本，越是复杂的应用，需要加载的JavaScript脚本就越多、越大，这会导致应用的首屏加载时间非常长，进而降低了体验感。

同构

所谓同构，就是让一份代码，既可以在服务端运行，也可以在客户端运行，并且执行的效果都是一样的，都是完成这个html的组装，正确的显示页面。
同构条件：需要实现客户端与服务端的路由、模型组件、数据模型的共享

技术栈

express + react + react-router + redux

Node服务器的安装

  2. 实现一个简单的页面。在根目录创建一个入口文件app.js，监听3000端口的请求，当请求根目录时，返回HTML

    Webpack的安装

    使用webpack，打包项目，解析react语法文件等

  3. 安装webpack npm install webpack webpack-cli --save

  4. 创建配置文件webpack.server.js

  5. 安装对应的babel

babel是一个工具链，主要用于将采用es2015+语法编写的代码转换为向后兼容的js语法，以便能够运行在当前和旧版本的浏览器或其他环境中。

使用renderToString渲染组件

因为服务端没有DOM，所以无法使用react语句，获取dom进行渲染。
所以需要使用react-dom提供的renderToString方法，将组件渲染成为字符串再插入返回给客户端的HTML中。

实现服务器自动重启和打包

安装nodemon，实现监听文件的变化并自动重启服务器
安装npm-run-all，实现一步启动项目自动打包和服务器重启

实现同构

因为renderToString只渲染了组件内容，而不会绑定事件，所以需要将react代码在服务端执行一遍，在客户端再执行一遍，也就是同构。

  1. 通过<script>标签为页面引入客户端执行的React代码，并通过express的static中间件为js文件配备路由，修改原来的app.js
  2. 编写客户端react代码
  3. 实现客户端的webpack配置文件，用于客户端代码的打包方式

    路由同构

    通过Router.js存放路由相关信息 在客户端，一般使用BrowserRouter来引入路由信息 而在服务端渲染时，需要用StaticRouter来替换BrowserRouter。因为BrowserRouter会通过HTML5提供的history API来保持页面与URL同步，而StaticRouter则不会改变URL。
    StaticRouter接受2个参数：

  • location：StaticRouter 会根据这个属性，自动匹配对应的React组件，所以才会实现刷新页面，服务端返回的对应路由的组与浏览器端保持一致

  • context：一般用来传递一些数据，相当于一个载体，可用于传递样式信息

    在服务端配置路由是为了能根据url匹配到对应组件内容，实现与客户端内容保持一致。而在客户端初次渲染完成后，客户端的路由将接管页面的路由，在之后的路由跳转中，实现无刷新跳转。

    数据同构

    在redux同构中需要实现数据的预获取以及脱水与注水

  • 脱水（Dehydrate）：在生成html之前，需要去请求接口获取数据，然后将数据塞到页面中去，然后将生成的html返回给客户端

  • 注水（Hydrate）：客户端拿到带着数据的HTML，去请求客户端的js，接管页面，然后用这个数据初始化组件

    组件的配置：

    因为服务端是不会执行componentDidMount钩子函数的，所以在服务端需要使用loadData来获取数据。
    loadData需要接受服务端的store，并通过store.dispatch来扩充服务端的store

    路由的改造

    改用数组存储路由配置信息，添加loadData参数，指定组件内loadData方法

    Store相关代码

    createStore创建的store是单例的store，在服务端会导致所有用户使用同一个store，所以需要封装一个store获取方法，每次获取store都通过createStore创建一个新的store返回

    客户端入口文件添加redux

    使用Provider组件注入store，并遍历路由配置信息，使用Route组件加载路由

    服务端渲染函数添加redux

    在服务端渲染中，因为获取数据是异步的，所以需要借用promise，在实现渲染与获取数据的先后顺序。使得执行完所有loadData后再执行渲染。
    使用matchPath根据当前path来匹配对应的路由，并使用promise来包裹loadData。防止其中一个promsie报错而导致其他promise无法执行
    最后的Promise.all的回调中执行渲染

    注入数据

    一个小优化的点：将html内容放置到html文件中，通过fs读取该文件内容
    从服务端的store中取出数据注入到要返回客户端的html中，

    store修改

    因为客户端还需要从window.context中获取初始数据，所以客户端和服务端需要单的的store入口文件

    数据统一的整个流程：

  • 用户访问路由，服务端根据路由匹配出对应路由内的组件数组

  • 循环数组，调用组件上挂载的loadData方法，发送请求，扩充服务端store

  • 所有请求完成后，通过store.getState，获取到服务端预获取的数据，注入到window.context中

  • 浏览器渲染返回的HTML，加载浏览器端js，从window.context中取数据来初始化浏览器端的store，渲染组件
